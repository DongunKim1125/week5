# Project Mandates (GEMINI.md)

이 파일은 Gemini CLI의 동작을 규정하는 핵심 지침을 담고 있습니다.

## 0. 대답 규칙
- 작업이 끝나면 어떤 작업을 했는지 요약
- 작업 후 유니티 인스펙터 설정을 변경할 것이 있다면 빠짐없이 모두 말할 것

## 1. 프로젝트 개요
- **유형:** Unity 2D 퍼즐 플랫포머 (Room-Shifting Puzzle)
- **핵심 메카닉:** 하나의 '타일'은 독립된 미니 레벨이며, 타일을 재배치하여 플레이어의 이동 경로를 생성함.
- **개발자:** Dongeon (프로젝트 내 'Dongeon' 폴더는 개발자 이름을 따름)

## 1. 게임 개요
유형: Unity 2D (퍼즐 플랫포머)
핵심 컨셉: 미니 레벨이 담긴 타일을 드래그하여 길을 만드는 'Room-Switching' 메커니즘

1. 핵심 메커니즘 (Core Hook)
- **미니 레벨 타일:** 각 타일은 내부에 플랫폼, 장애물, 상호작용 오브젝트를 포함한 독립된 공간임.
- **타일 이동:** 플레이어가 점유하지 않은 타일(방)은 5x4 그리드 내에서 드래그하여 위치를 바꿀 수 있음.
- **연속성:** 인접한 타일끼리는 경계를 통해 플레이어가 넘나들 수 있음.

2. 주요 게임플레이 규칙
- **점유 제약:** 플레이어가 현재 위치한 타일은 드래그로 옮길 수 없음 (자아 구속적 제약).
- **낙사(Void Death):** 타일의 경계를 벗어났을 때 해당 위치에 타일이 없으면 플레이어는 즉시 사망함.
- **승리 조건:** 시작 타일에서 출발하여 목표(Goal) 타일의 특정 지점에 도달해야 함.

3. 시스템 아키텍처
- **GridManager:** 5x4 그리드에서 각 '방(Tile)'의 좌표 데이터를 관리.
- **Tile:** 타일 내부의 월드 크기(Size) 정보를 가지며, 플레이어의 진입/퇴장을 감지함.
- **DeathZone:** 타일이 없는 그리드 공간은 물리적/논리적 데드존으로 작동함.

## 2. 코딩 규칙

명명 규칙: - C# PascalCase: 클래스, 메서드, public 변수, 속성(Property)

camelCase: private/protected 변수(접두사 _ 권장), 매개변수, 지역 변수

주석: 한글로 작성하며, 복잡한 로직이나 클래스 상단에는 XML 문서 주석(///)을 필수 적용함.

Unity API 성능 최적화:

Update 내에서 GetComponent, Find, SendMessage 호출을 엄격히 금지함.

컴포넌트 참조는 Awake 또는 Start에서 캐싱하거나, 최신 API인 TryGetComponent를 사용함.

모든 타일 상호작용은 물리 엔진 기반(OnTriggerEnter2D)으로 처리함.

## 3. 도구 및 워크플로우

협업 구조: - System Manager (User): 그리드 데이터 관리, 타일 이동 로직, 전체 게임 흐름 담당

Actor Engineer (Partner): 플레이어 조작, 오브젝트 기믹 효과 담당

Git: GitHub Desktop을 사용하며, 기능 단위로 브랜치를 생성하여 작업함. 커밋 메시지는 [Feat], [Fix], [Refactor] 등의 접두사를 사용함.

Gemini CLI 활용: 코드 생성 시 기존 GridManager의 싱글톤 인스턴스를 참조하여 의존성을 해결함.

## 4. 핵심 시스템 설계 (Architectural Mandates)Gemini CLI는 다음 규칙에 기반하여 코드를 생성해야 합니다.

그리드 제약: - 기본 배경 그리드는 5 * 4 크기이며, 모든 좌표는 Vector2Int로 관리함.

시작(Start) 타일과 목표(Goal) 타일은 isMovable = false 상태여야 함.

이동 제약: - Tile.isOccupiedByPlayer가 true일 경우, 해당 타일의 드래그 로직은 무시되어야 함.

타일은 다른 타일과 겹칠 수 없으며, 빈 그리드 칸(null인 좌표)으로만 이동 가능함.

생존 판정: - 플레이어는 매 프레임 발밑의 타일 레이어를 확인하며, 타일이 없을 경우 즉시 사망 시퀀스를 실행함.
